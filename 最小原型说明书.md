## 最小原型（MVP）制作与技术实现说明书
版本：1.0

核心目标：以最低成本、最快速度实现一个可运行的桌面级AI二次元互动硬件原型，验证核心功能与技术可行性。

## 1. MVP概述
### 1.1 产品定义
名称：次元萌盒（原型代号）

形态：横屏长方体（25cm × 10cm × 18cm），桌面放置

核心定位：沉浸式二次元陪伴，能看、能听、能说、能陪你“闹腾”的AI动漫伙伴

### 1.2 MVP功能范围
角色：选定《火影忍者》漩涡鸣人

对话：支持日常闲聊、剧情接梗、经典台词，上下文记忆≥5轮

手势彩蛋：3个鸣人经典手势（螺旋丸、结印、打招呼），识别触发台词/音效

人设定制：可切换软萌 / 中二两种互动风格

记忆系统：记录用户偏好（常用话题、触发彩蛋），用于风格适配

硬件：10.1寸触控屏、摄像头、扬声器、麦克风、锂电池，3D打印外壳

### 1.3 MVP验证指标
| 维度 | 指标 | 合格标准 |
|------|------|----------|
| 功能 | 基础对话 | 无OOC，上下文记忆≥5轮 |
| 功能 | 手势识别 | 3个手势准确率≥85%，触发延迟≤1秒 |
| 功能 | 人设定制 | 风格切换有效，语气区分明显 |
| 技术 | 对话延迟 | 用户输入→语音回应≤3秒 |
| 技术 | 语音还原 | 声纹相似度≥80%，清晰无杂音 |
| 硬件 | 稳定性 | 连续运行4小时无过热/死机 |
| 硬件 | 续航 | 满电连续使用≥4小时 |

## 4. AI模型训练与部署（核心）
### 4.1 沉浸式对话（LLM）实现
目标：让鸣人的对话风格、语气高度还原，支持上下文记忆。

#### 4.1.1 语料库构建
1. 收集鸣人原版台词（动画、游戏）≥3万条，覆盖日常、战斗、羁绊等场景
2. 清洗OOC内容（脱离角色设定的语句）
3. 格式：每行一条文本，可加入情感/场景标签（如 [日常]、[战斗]）

#### 4.1.2 模型选型与量化
- 基座模型：Qwen-7B 或 Llama2-7B（中文支持好）
- 量化：使用AutoGPTQ或GPTQ-for-LLaMA将模型4bit量化，减小体积至约4GB，适配RK3568内存
- 也可使用llama.cpp的GGUF格式，在ARM上高效运行

#### 4.1.3 LoRA微调
- 在GPU服务器上进行LoRA微调（如使用peft库）
- 参数：r=8，lora_alpha=32，学习率5e-4，训练3~5个epoch
- 语料注入后保存LoRA权重（约几十MB）

#### 4.1.4 部署到RK3568
- 方案A：使用llama.cpp加载GGUF模型，提供HTTP接口或C++调用
- 方案B：使用AutoGPTQ加载量化模型，通过Python服务调用
- 内存优化：模型加载后常驻内存，对话时仅推理，控制最大生成长度（128 tokens）

### 4.2 手势识别（CV）实现
目标：识别用户做出的3个鸣人经典手势，触发彩蛋。

#### 4.2.1 手势特征库构建
1. 基于MediaPipe Hands（支持21个关键点）

4. 设定匹配阈值（欧氏距离或余弦相似度≥85%）

### 4.3 原声复刻（TTS）实现
**目标**：将 LLM 生成的文本转换为鸣人原声。

#### 4.3.1 音频采集
- 收集鸣人原版配音（动画、游戏）≥1 小时，覆盖不同情绪（普通、激动、温柔）
- 切割为 5~15 秒的短音频，标注文本
- 格式：16kHz 采样率，单声道，WAV

#### 4.3.2 模型训练
- 使用 VITS 或 Coqui TTS（中文模型）
- 在 GPU 服务器上进行微调，需提供音频 - 文本对
- 训练至损失收敛，合成测试音频，确保声纹相似度≥80%

#### 4.3.3 部署推理
- 将训练好的 TTS 模型转换为 TorchScript 或 ONNX，减小体积
- 在 RK3568 上运行 Python 推理，生成语音后立即播放
- 缓存常用回复，减少实时生成频率

### 4.4 AI 记忆系统实现
**目标**：记录用户偏好，实现人设定制与个性化回应。

#### 4.4.1 数据库设计
- 使用 SQLite 本地数据库
- 表：user_preferences（用户 ID、话题频率、触发彩蛋次数、常用风格）
- 表：conversation_history（最近 20 轮对话，用于上下文）

#### 4.4.2 人设定制逻辑
- 界面提供 “软萌”、“中二” 两个按钮
- 点击后，将风格标签（如 style: tsundere）附加到 LLM 的 system prompt 中
- 示例 prompt：“你现在是漩涡鸣人，性格【中二】，说话热血、充满干劲。”

#### 4.4.3 个性化优化
- 每完成一次对话，记录话题关键词（如 “修行”、“拉面”）
- 当某个话题出现频次超过阈值，在后续对话中主动提及
- 可定期（如每周）重新生成少量微调数据，但 MVP 阶段暂不自动重训

## 5. 软件集成与界面开发
### 5.1 简易交互界面（Python + PyQt / Kivy / Android App）
- 主界面：角色立绘（静态图），下方对话气泡显示区域
- 输入方式：语音按钮（按住说话）或文字输入框
- 手势引导：显示当前支持的 3 个手势图示
- 人设切换：两个风格按钮，点击后弹出提示
- 设置：灵敏度调节、音量、清除记忆

### 5.2 后端服务整合
编写一个主控 Python 脚本（或 Android Service），负责：
- 监听语音输入（调用 speech_recognition 或离线识别）
- 调用 LLM 生成回复
- 将文本送入 TTS 合成并播放
- 实时读取摄像头帧进行手势识别
- 记录偏好到 SQLite
- 多线程 / 异步处理，避免阻塞 UI

### 5.3 端到端流程示例
用户点击 “按住说话” 按钮，麦克风录音 → 离线语音识别（如 Vosk）得到文本 → 文本 + 当前风格 + 最近 5 轮对话 → 发送给 LLM 服务 → 得到回复文本 → 回复文本送入 TTS 生成音频 → 播放
同时摄像头持续运行，若识别到手势，立即播放彩蛋音效并显示动画
所有交互数据存入 SQLite

### 6.1 分模块测试

- 记忆：切换风格后检查回复变化，重启后偏好是否保留

### 6.2 全流程压力测试
- 连续进行 100 轮对话，记录 OOC 次数、延迟超标次数
- 连续触发手势 50 次，统计误判 / 漏判
- 整机运行 4 小时，监测温度（≤60℃）和系统流畅度

### 6.3 问题修复优先级
- P0：无法对话、手势完全不识别 → 立即修复
- P1：延迟过高、语音失真 → 优化模型或量化
- P2：UI 卡顿、小 bug → 后续迭代

## 8. 常见问题与注意事项
- 模型体积过大：务必使用 4bit 量化，并只加载必要的层。RK3568 内存有限，可考虑使用 llama.cpp 的 mmap 映射。
- 手势识别光照敏感：测试时保持室内光线充足，必要时用补光灯。
- 语音识别准确率：离线识别可能不完美，MVP 阶段可先用按键输入文字代替语音，降低复杂度。
- 散热：RK3568 满载时发热明显，必须安装散热片，并保证通风。
- 数据备份：语料库、模型权重、手势特征库均需多处备份。
- 安全：锂电池充电时需有人值守，避免短路。
